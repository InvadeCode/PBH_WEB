01 System Architecture Overview

The entire Scope Builder is a single React component called StrategicEngine, defined around line 923 of src/App.jsx. It is rendered at the /assessment route and receives one prop — navigate — which is the app's internal page-switching function.

Component State

All logic is driven by a flat state object inside the component. There is no external state manager (Redux, Zustand, etc.). Every variable is a React useState hook:

03 Step Machine — Complete Phase Flow

The step machine is an array called stepsArray. Rendering is done by indexing into it: stepsArray[step]. AnimatePresence mode="wait" wraps each step so there is a slide transition between every state change.

With 6 quiz questions (N = 6), the full step index map is:

0 — Splash → 1–6 — Quiz → 7 — Diagnosis → 8 — Route Select → 9 — Deliverables → 10 — Context → 11 — Lead Capture → 12 — Scope Snapshot
0

Splash — Brand Identification

UI: Two text inputs: "Brand / Business Name" (required, maps to leadForm.company) and "Industry / Sector" (optional, maps to leadForm.industry).

Gate: "Start Scope Builder" button is disabled if leadForm.company is empty. On click: setStep(1).

Note: This is the only step that does not show the LiveScopePreview sidebar. The sidebar appears for steps 1 through N+5 only.

1–6

Phase 1 — Diagnosis (Quiz Questions)

Source: Generated dynamically via QUIZ_QUESTIONS.map(). One step per question.

Single-select behavior: Clicking an option fires handleQuizSelect(), which stores the answer, then automatically advances to the next step after a 500 ms delay (giving the selection a visual moment before transitioning). On the final question it calls processDiagnosis() instead of advancing.

Multi-select behavior: Options toggle on/off. A "Select All" button selects every non-none option. Advancing requires a "Continue" button click, calling handleMultiSelectContinue(). If all questions have been answered (step === N), it calls processDiagnosis(); otherwise it increments step.

Comments field: Each question has a textarea below the options. Values are stored in comments[questionId]. These appear in the PDF and Excel output.

Progress bar: A fixed top bar appears (steps 1–N+5) showing phase name, percentage, animated fill, and a pulsing beacon dot at the progress edge.

N+1

Diagnosis Result — Opportunity Areas

Trigger: Reached automatically after the last quiz answer is processed by processDiagnosis(), which calls setStep(N_QUIZ + 1).

UI: Displays the identified clusters as a natural language summary, then shows route cards for each recommended route. Each card has a "Read More" toggle that expands a multi-paragraph expandedDesc narrative.

Action: "Customize Scope" advances to N+2.

N+2

Route Selection — Scope Architecture

UI: Checkboxes for all three routes (BB, SAS, STC). Pre-selected based on routes from diagnosis. User can add or remove any route.

Gate: "Select Deliverables" disabled unless at least one route is selected.

State written: selectedRoutes

N+3

Deliverables Selection — Build Your Scope

UI: Renders only the line items and deliverables belonging to selectedRoutes. Each deliverable is a checkbox card. Selected cards show a Priority dropdown: Standard | High | Critical.

Dependency guard: Toggling a deliverable passes through handleDeliverableToggle(), which checks for unmet prerequisites or broken dependents and shows a modal if any are found. See Section 5.

"Select All" per line item: Adds all deliverables in that line item to selectedDeliverables at once, bypassing the dependency modal.

State written: selectedDeliverables, priorities, warnings

N+4

Project Context — Execution Parameters

UI: Three <select> dropdowns.

Depth: Light-touch consulting / Strategic direction + selected assets / End-to-end brand transformation

Timeline: Short term / Deep Dive / Long Term

Duration: Short term (min 3 months) / Deep Dive- Branding (min 6 months) / Long Term Engagement (1 year)

Gate: "Finalize Blueprint" disabled unless both Depth and Timeline are selected. Duration has a default of "Deep Dive - Branding."

State written: context.depth, context.timeline, context.duration

N+5

Lead Capture — Contact Details

UI: Three required inputs: Full Name, Work Email, Phone Number. The company name was already collected at Step 0.

Submit: Fires submitLead(e), which generates the PDF, Excel, sends emails, saves to Supabase, then calls setStep(N_QUIZ + 6) on success. See Section 8.

Error: On failure, alert() shows the Resend API error. State remains at N+5 so the user can retry.

N+6

Official Scope Snapshot — Final Output

The read-only report rendered in the browser. The progress bar and sidebar are hidden at this step. Print/PDF action is available. See Section 7.

Back Navigation Every step (except 0 and N+6) has a top-bar "Back" button that simply calls setStep(step - 1). There is no validation on back navigation — the user can freely move backwards without losing any selections.

04 Diagnostic Engine

The diagnosis logic is entirely contained in processDiagnosis(finalAnswers), called once after the final quiz answer is submitted. It operates in three passes:

Pass 1 — Cluster Scoring

const clusterScores = {};
Object.values(finalAnswers).forEach(ans => {
  const opts = Array.isArray(ans) ? ans : [ans]; // normalize single → array
  opts.forEach(opt => {
    if (opt && opt.cluster) {
      clusterScores[opt.cluster] = (clusterScores[opt.cluster] || 0) + (opt.weight || 1);
    }
  });
});

Each selected option accumulates its weight value into the cluster it belongs to. Options with cluster: 'None' or weight 0 contribute nothing. After this loop, clusterScores is a dictionary like { 'Messaging Inconsistency': 4, 'Generic Identity': 2 }.

Pass 2 — Winner Selection

let maxScore = 0, topClusters = [];
for (const [cluster, score] of Object.entries(clusterScores)) {
  if (score > maxScore)      { maxScore = score; topClusters = [cluster]; }
  else if (score === maxScore) { topClusters.push(cluster); }  // tie → include both
}

The engine finds the highest scoring cluster. In case of a tie (two clusters both hit the max), all tied clusters are included as co-winners. This means the user can receive multiple route recommendations.

Pass 3 — Route Assignment

const foundRoutes = new Set();
topClusters.forEach(c => {
  if (clusterToRoute[c]) foundRoutes.add(clusterToRoute[c]);
});
const recRoutes = Array.from(foundRoutes).length > 0 ? Array.from(foundRoutes) : ['BB'];
setRoutes(recRoutes);
setSelectedRoutes(recRoutes); // pre-tick the route selection checkboxes
setStep(N_QUIZ + 1);
Example Diagnosis A user who selects "Different teams communicate differently" (cluster: Messaging Inconsistency, weight: 2) on Q1, and "Messaging inconsistency across touchpoints" (weight: 2) on Q5, accumulates score 4 for Messaging Inconsistency. If no other cluster scores 4 or higher, the result is ['BB'] → Brand Boulevard recommended.

Why clusters are stored (not just routes)

The clusters state is used in three downstream places: (1) the BrandHealthRadar SVG visualization lowers specific dimension scores based on which cluster is active; (2) it appears in the email HTML sent to the PBH internal team; (3) it is saved to Supabase for CRM analysis.

05 Dependency System

The dependency system prevents a user from selecting an advanced deliverable without its prerequisites, and warns when removing a deliverable that other selected deliverables depend on. It uses a Breadth-First Search (BFS) graph traversal.

parsePrerequisites(field)

Takes the raw interdependence string and returns a flat array of prerequisite IDs:

const parsePrerequisites = (field) => {
  if (field === "None" || !field) return [];
  const results = [];
  const parts = field.split(",");
  for (let part of parts) {
    part = part.trim();
    const rangeMatch = part.match(/^D(\d{3})[–-]D(\d{3})$/);
    if (rangeMatch) {
      // e.g. "D019-D026" → ['D019','D020',...,'D026']
      const start = parseInt(rangeMatch[1]), end = parseInt(rangeMatch[2]);
      for (let n = start; n <= end; n++) results.push("D" + String(n).padStart(3, "0"));
    } else if (part.match(/^D\d{3}$/)) {
      results.push(part);
    }
  }
  return results;
};
Important: The regex handles both regular hyphens (-) and em-dashes (–) in the range syntax, since copy-paste from design documents may introduce either character.

findUnmetPrerequisites(deliverableId, currentSelections, deliverablesList)

Returns every prerequisite of deliverableId (and their transitive prerequisites) that are not already in currentSelections. Uses iterative BFS to avoid stack overflows on deep chains:

const findUnmetPrerequisites = (deliverableId, currentSelections, deliverablesList) => {
  const unmet = [], queue = [deliverableId], visited = new Set();
  while (queue.length > 0) {
    const current = queue.pop();
    if (visited.has(current)) continue;
    visited.add(current);
    const deliv = deliverablesList.find(d => d.id === current);
    if (!deliv || deliv.interdependence === "None") continue;
    const prereqs = parsePrerequisites(deliv.interdependence);
    for (const prereq of prereqs) {
      if (!currentSelections.includes(prereq) && !unmet.includes(prereq)) {
        unmet.push(prereq);
        queue.push(prereq); // recurse into this prereq's own prereqs
      }
    }
  }
  return unmet;
};

findAllDependents(deliverableId, currentSelections, deliverablesList)

Reverse lookup: finds every deliverable in currentSelections that would be broken (i.e., have an unmet prerequisite) if deliverableId were removed:

const findAllDependents = (deliverableId, currentSelections, deliverablesList) => {
  const dependents = [];
  for (const sel of currentSelections) {
    if (sel === deliverableId) continue;
    // Simulate removal by passing selections without deliverableId
    const unmet = findUnmetPrerequisites(sel, currentSelections.filter(x => x !== deliverableId));
    if (unmet.includes(deliverableId)) dependents.push(sel);
  }
  return dependents;
};

handleDeliverableToggle(dId) — The Guard Gate

This is the function called whenever a user clicks a deliverable checkbox:

ScenarioAction
Adding a deliverable with no unmet prerequisites Directly adds to selectedDeliverables. No modal.
Adding a deliverable with unmet prerequisites Opens dependencyModal with type: 'add'. Lists all missing prereqs. User can: (A) add all prerequisites automatically, or (B) add the item anyway with a warning badge.
Removing a deliverable with no dependents Directly removes from selectedDeliverables and clears any warning for it.
Removing a deliverable that others depend on Opens dependencyModal with type: 'remove'. Lists all dependents that would break. User can: (A) remove both the item and all its dependents, or (B) cancel.
Warning State When a user chooses "Keep with warning" on add, the deliverable ID is added to warnings[]. The deliverable card displays a yellow "Missing Prereqs" badge. This warning is visible in the Scope Snapshot in the browser and in the PDF/Excel output via a ⚠️ icon next to the deliverable name.

computeSuggestedStartingPoint(selectedDelivs, priorities, deliverablesList)

Called at submission time to determine what the client should work on first:

  1. Expand the selected deliverables with all their transitive prerequisites.
  2. Filter this expanded set to only foundation deliverables (the layer01Ids set).
  3. If none are found, return the name of the first deliverable in the set.
  4. If exactly one foundation deliverable is found, return its name.
  5. If multiple foundations exist, return the name of the first one (simplified resolution for the current build).

The result appears in the Scope Snapshot's Execution Context table as "Suggested Starting Point" and in the PDF's Strategy Blueprint section.

calculateCompleteness()

A simple additive score — not a percentage of required fields completed, but a readiness signal:

const calculateCompleteness = () => {
  let score = 50;                             // baseline: always at least 50%
  if (answers.stage)              score += 20; // brand stage answered
  if (clusters.length > 0)        score += 15; // diagnosis completed
  if (selectedDeliverables.length > 0) score += 15; // at least one deliverable chosen
  return score; // max: 100%
};

The score drives a SVG circle stroke animation. The stroke color switches from indigo (palette.blue) to green (palette.green) when the score reaches 100.

Panel animations

The Active Directives list uses Framer Motion's layout prop with AnimatePresence, so deliverables animate in (slide from left) and out (slide to right) individually as the user toggles them. Each new item's checkmark SVG path draws with pathLength: 0 → 1 on entry. Each node on the connection spine pulses with a scale oscillation on a 2s loop, color-coded to the panel it belongs to.

08 Submit Flow & Output Generation

When the user submits the lead form at Step N+5, submitLead(e) runs a multi-stage async pipeline:

1

Lead Data Assembly

Builds leadData: merges leadForm with clusters, routes, deliverables, context, startingPoint, current ISO date, status 'New', and a random score (60–99) as a placeholder engagement score.

Groups deliverables by Route → LineItem for clean PDF/email structure.

2

PDF Generation (jsPDF)

Uses jsPDF (A4 pt units) to generate a multi-page PDF entirely in the browser:

  • Dark header block (#121228) with "Brand Scope Report" title and confidential label
  • Client details section
  • Diagnostic Responses: each question in a rounded rectangle block with question title (bold), selected answer(s) (indigo bullet), and any comments
  • Strategy Blueprint: Depth, Timeline, Suggested Starting Point in key/value pairs
  • Selected Deliverables: grouped by Route then LineItem, rendered in a 2-column grid with individual deliverable names
  • Footer on each page: "PurpleBlue House — Scope Builder Report"
  • checkPage() is called before every block — if the content would overflow the page bottom margin, it inserts a new page first.

Output: Base64-encoded via doc.output('datauristring'), then split to extract raw base64.

3

Excel Generation (ExcelJS)

Uses ExcelJS to build a formatted workbook in the browser. The single worksheet "Scope of Work" includes:

  • PBH branding header row with dark fill and white bold text
  • Company, contact, and submission date rows
  • Diagnostic Responses section header + one row per answered question
  • Selected Deliverables section with ID, name, priority, and interdependence for each item
  • Execution Context rows (Depth, Timeline, Duration)

Output: Written to an ArrayBuffer, converted to binary string, then base64-encoded via window.btoa(binary).

4

Parallel Async Operations

Three promises are fired simultaneously via Promise.all():

  • saveLeadPromise → POST /api/save-lead with the full lead data object. This endpoint saves to Supabase (or Sanity). Failure here is logged as a warning but does not block the user flow.
  • emailPromise → sendEmailViaResend(subject, htmlContent, attachments) — sends to the PBH internal team (anant@invadecode.com). Attachments: PDF + Excel.
  • clientEmailPromise → sendEmailViaResend(clientSubject, clientHtmlContent, clientAttachments, leadForm.email) — sends to the respondent. Attachments: PDF only (no Excel).
5

Step Advance

If the internal email promise resolves with success: true, calls setStep(N_QUIZ + 6) to show the Scope Snapshot screen. If it fails, shows an alert() with the Resend API error and stays on Step N+5.

sendEmailViaResend

A shared async helper at the top of App.jsx. It POSTs to /api/send-email with a JSON body containing to, subject, html, and attachments. The server-side handler calls the Resend API. The default to address (when no toEmail arg is passed) is hardcoded to the PBH internal address.

09 State Persistence — LocalStorage

To prevent users from losing progress on accidental page refresh, the entire StrategicEngine state is persisted to localStorage on every change and restored on mount.

Key

const STORAGE_KEY = 'pbh_scope_builder_state';

What is Persisted

{ step, answers, comments, clusters, routes, selectedRoutes,
  selectedDeliverables, priorities, context, leadForm }

The warnings array, expandedRoute, dependencyModal, and isSubmitting are deliberately not persisted — these are transient UI state that should reset on page load.

Restore on Mount

useEffect(() => {
  const savedState = localStorage.getItem('pbh_scope_builder_state');
  if (savedState) {
    try {
      const parsed = JSON.parse(savedState);
      // Each key is individually restored via its setter,
      // guarded by an existence check to avoid overwriting defaults.
      if (parsed.step) setStep(parsed.step);
      if (parsed.answers) setAnswers(parsed.answers);
      // ... and so on for each key
    } catch (e) {
      console.error("Failed to load saved progress", e);
    }
  }
  setIsLoaded(true); // gate: prevents write-effect from firing before restore
}, []);

Write Effect

useEffect(() => {
  if (isLoaded) {
    localStorage.setItem('pbh_scope_builder_state', JSON.stringify({
      step, answers, comments, clusters, routes, selectedRoutes,
      selectedDeliverables, priorities, context, leadForm
    }));
  }
}, [step, answers, comments, clusters, routes, selectedRoutes,
    selectedDeliverables, priorities, context, leadForm, isLoaded]);
Clearing Saved State There is currently no "Start Over" or "Clear Progress" button in the UI. To reset, a developer can run localStorage.removeItem('pbh_scope_builder_state') in the browser console. A user clearing their browser data will also clear the state.
